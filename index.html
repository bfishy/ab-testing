A/B Testing

Performing A/B testing by observing user behaviors

Website Overview

Cactus Shopping Website

Link Here

The website resembles a simple shopping site where users can add items (cacti) into their shopping cart.

Each cactus has a price & the number of each cactus added in the shopping cart is shown next to the "Add to Cart" Button

The shopping cart icon also shows the total number of items added

When the user clicks the cart, they would be redirected to a checkout page, where the total cost would be shown

Data Collection Process:

There are two versions to this website, Version A & B.

Users that visit the website will be randomly redirected to one of the two, and their behavior will be logged.

Logs are used data such as when the users entered the site & when they left, which version of the site was used, when/which items they clicked on & the time they entered the checkout page and/or left.

Users in this projects are fellow students who are asked to perform the task of adding $150 worth of cacti into their shopping cart & checking out.

We then compare between the 2 versions using two metrics: Time to Completion & Return Rate.

Time to Completion: The time it takes to complete the desired task
Return Rate: The number of times a user returned to the page

Hypotheses

Time to Completion

Null Hypotheses

Version A & B perfoms the same

- Users take the same amount of time to complete the task in both versions.

Alternative Hypotheses

Version B performs worse than Version A

- Users take longer to complete the task in Version B

Return Rate

Null Hypotheses

Version A & B perfoms the same

- Users return to the shopping page from checkout the same number of times in both versions.

Alternative Hypotheses

Version B performs better than Version A

- Users are less likely to return to the shopping page after checkout out in Version B

Discussion & Follow-Up

  • The results & statistics of this project was primarily impacted by a website bug in Version B.

    As a result, the logs for Version B, initially, had really high Time to Completion & really low Return Rate since users were confused by the bug & weren't able to "return" since they couldn't access the checkout page.

    I invited some new users to try out the new version after fixing the bug, and made my conclusions on accepting/rejecting the hypotheses based on the collected data.

  • The fact that, given the above bug, average Time to Completion between the 2 versions did NOT differ by a lot suggests that the design changes still had a positive effect.

  • By applying principles like proximity in version B & re-positioning the "Add to Cart" button, users are able to see the item, the price, and the button as one action group, making it easier to check the price & buy the item immediately.

    In contrast, user of Version A have to read all the way to the end of the product description to know how much they'd have to pay.

  • By adjusting font sizes, we can also create an importance hierarchy.

    When the price is shown in larger font, it stands out from the descriptions & is more noticeable.

    This is good because price is one of the main determining factors for purchasing an item.
